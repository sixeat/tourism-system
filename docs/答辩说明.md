# 智旅旅行系统答辩说明

## 1. 项目简介

本项目是一个旅游推荐与预订系统，名称为“智旅旅行”。系统围绕游客出行前的核心需求展开，包括目的地推荐、路线规划、酒店预订、景点门票、订单管理、AI 问答和地图展示。

系统采用 Java SSM 架构开发，前端使用 JSP 页面，后端使用 Spring、Spring MVC、MyBatis，数据库使用 MySQL，部署在 Tomcat 服务器上。

## 2. 选题背景

旅游平台通常需要同时解决路线、住宿、门票和订单管理等问题。传统旅游网站功能较分散，用户需要在多个页面之间切换。本项目希望通过一个综合系统，把路线推荐、酒店门票预订、地图展示和智能问答整合到同一个平台中，提高用户规划出行的效率。

## 3. 系统目标

1. 为用户提供旅游路线推荐
2. 支持酒店和景点门票预订
3. 支持订单创建、支付、取消和评价
4. 支持高德地图展示路线和景点
5. 支持 AI 旅游问答
6. 为管理员提供资源维护和订单管理能力

## 4. 系统架构

系统分为表现层、控制层、业务层、数据访问层和数据库层。

1. 表现层：JSP 页面，包括首页、登录页、订单页、后台管理页
2. 控制层：Spring MVC Controller，负责接收请求并返回 JSON 数据
3. 业务层：Service，负责登录、推荐、下单、支付、取消、AI 问答等业务逻辑
4. 数据访问层：MyBatis Mapper，负责操作 MySQL 数据库
5. 数据库层：MySQL，保存用户、景点、酒店、门票、路线和订单数据

## 5. 核心模块讲解

### 5.1 用户模块

用户可以注册新账号，也可以使用默认账号登录。登录成功后系统通过 Session 保存用户状态。

演示接口：

1. `POST /api/auth/login`
2. `POST /api/auth/register`
3. `GET /api/auth/me`

### 5.2 路线推荐模块

用户在首页输入目的地、预算、天数和兴趣关键词后，系统会推荐匹配的旅游路线。

路线详情中展示：

1. 推荐路线名称
2. 目的地城市
3. 游玩天数
4. 预计费用
5. 景点游览顺序
6. 每日行程安排
7. 酒店、门票、交通餐饮和机动预算拆分

答辩讲解重点：

路线推荐不是只返回一个名称，而是进一步拆成每日计划和预算明细，更贴近真实旅游产品。

### 5.3 酒店与门票模块

系统支持按城市查询酒店，查看房型和库存；也支持查看景点门票，选择门票后可以直接下单。

答辩讲解重点：

酒店、门票和快速下单区域是联动的，用户点击“选择房型”或“选择门票”后，订单表单会自动填充编号。

### 5.4 订单模块

订单模块是项目的业务闭环部分。用户可以创建酒店订单和门票订单，订单状态初始为 `CREATED`。

订单支持：

1. 取消订单：`CREATED` 状态订单可取消
2. 模拟支付：`CREATED` 状态订单支付后变为 `PAID`
3. 订单评价：已支付订单可填写评分和评价
4. 订单筛选：支持按订单类型和订单状态筛选

答辩讲解重点：

订单流程从“创建订单”到“支付订单”再到“评价订单”，形成完整闭环。

### 5.5 AI 问答模块

系统首页提供 AI 旅游问答功能。AI 问答使用本地 RAG 检索方式，会先从系统中的景点、酒店、门票、路线和订单规则中检索相关资料，再生成回答。

可演示问题：

1. 杭州三日路线怎么安排？
2. 西湖门票多少钱？
3. 杭州有哪些酒店可以订？
4. 订单创建后怎么取消？

答辩讲解重点：

AI 不是完全凭空回答，而是基于系统已有数据进行检索增强回答，所以答案能结合当前项目数据。

### 5.6 高德地图模块

首页接入高德地图，用于展示景点位置，并根据路线顺序绘制游览路径。

答辩讲解重点：

路线推荐和地图展示是联动的，点击不同路线可以同步地图上的景点顺序。

### 5.7 后台管理模块

管理员登录后可以进入后台管理页面，维护系统资源。

后台支持：

1. 景点管理
2. 酒店管理
3. 门票管理
4. 路线管理
5. 订单管理
6. 首页统计

答辩讲解重点：

前台展示的数据不是写死的，后台可以维护基础资源。

## 7. 数据库设计说明

系统核心表包括：

1. `sys_user`：用户信息
2. `scenic_spot`：景点信息
3. `hotel`：酒店信息
4. `hotel_room`：酒店房型
5. `ticket`：门票信息
6. `travel_route`：路线信息
7. `route_spot`：路线和景点关联
8. `hotel_order`：酒店订单
9. `ticket_order`：门票订单

其中，订单表通过 `order_status` 字段区分订单状态，如 `CREATED`、`PAID`、`CANCELLED`。

## 8. 项目亮点总结

1. 功能闭环完整：注册、登录、推荐、预订、支付、评价、后台管理
2. 推荐展示完整：路线卡片、每日安排、费用明细、地图联动
3. 引入 AI 问答：使用 RAG 思路结合系统数据回答问题
4. 页面中文化：界面文字统一中文，更适合课堂展示
5. 高德地图接入：将路线和景点可视化
6. 后台可维护：景点、酒店、门票、路线和订单都可管理

## 9. 当前不足

1. 收藏和评价已支持数据库持久化，但还可以继续增加审核、统计和推荐分析
2. 支付为模拟支付，未接入真实支付平台
3. 城市和景点数据量还可以继续扩充
4. AI 外部接口已预留，但当前主要使用本地 RAG 兜底

## 10. 后续优化方向

1. 增加评价审核、收藏统计和用户画像推荐
2. 增加更多城市、景点、酒店和门票数据
3. 接入真实支付或沙箱支付
4. 增加用户资料编辑、头像上传和密码修改
5. 接入正式大模型 API，提高 AI 问答质量
6. 将 JSP 前端升级为 Vue 或 React 前端

## 11. 答辩结束语参考

本项目围绕旅游出行场景，实现了从路线推荐、酒店门票预订、订单支付评价，到后台管理和 AI 问答的完整流程。系统虽然还有继续优化空间，但已经具备旅游平台的核心业务闭环，能够较好地展示 Java SSM 项目的分层设计、数据库操作、前后端交互和综合应用能力。
